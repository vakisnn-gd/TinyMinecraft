# План разработки

Главное правило: сначала стабилизация, потом большие новые функции. Snapshot 7 доказал, что LAN-мультиплеер уже можно запустить, но теперь его нужно сделать надежным, наблюдаемым и честно ограниченным.

## Цель v0.2 Snapshot 7

- Держать проект компилируемым на Java 8+.
- Держать одиночную игру запускаемой.
- Дать двум копиям игры подключиться через Direct IP/LAN.
- Сделать хост авторитетным по миру, блокам, чату и времени.
- Передавать клиенту чанки, изменения блоков, состояние игроков, мобов и дропа.
- Не добавлять внешние аккаунты, сторонние сетевые библиотеки и публичную инфраструктуру.

## Следующий проход стабилизации

1. Диагностика мультиплеера
   - Добавить явные console-логи для `HELLO`, `WELCOME`, `CHUNK_REQUEST`, `CHUNK_DATA`, `BLOCK_ACTION` и `BLOCK_UPDATE`.
   - Показывать в debug overlay сетевой режим, ping/RTT, количество remote players и очередь чанков.
   - Улучшить сообщения об ошибках при несовместимой версии протокола, занятом порте и duplicate UUID.

2. Стабильность подключения
   - Аккуратнее обрабатывать disconnect и повторное подключение.
   - Проверить закрытие сокетов и потоков при выходе в меню.
   - Защититься от зависания клиента при неполном handshake.
   - Добавить timeout для ожидания первых пакетов.

3. Синхронизация мира
   - Проверить корректность `CHUNK_DATA` на дальних координатах и отрицательных чанках.
   - Улучшить очередь запросов чанков, чтобы клиент быстрее догружал видимую область.
   - Снизить лишние повторные запросы колонок.
   - Проверить `BLOCK_UPDATE` для соседних блоков, жидкостей, прозрачности и пересборки мешей.

4. Сундуки и контейнеры
   - Спроектировать серверную модель контейнеров.
   - Добавить сетевые пакеты открытия, закрытия и изменения содержимого сундука.
   - Запретить клиенту напрямую менять авторитетное содержимое контейнеров.

5. Инвентарь и игровые действия
   - Разделить локальные UI-действия и серверные игровые действия.
   - Синхронизировать выбранный слот, выдачу предметов и расход блоков.
   - Уточнить поведение команд в multiplayer-сессии.

6. Минимальная защита протокола
   - Валидировать координаты и частоту `BLOCK_ACTION`.
   - Ограничить размер и частоту сетевых пакетов.
   - Проверять допустимую дистанцию действия с блоком.
   - Не считать это полноценным anti-cheat для публичных серверов.

## Идеи для следующих snapshot

- Dedicated server без OpenGL-клиента.
- Список локальных LAN-сессий через broadcast/discovery.
- Более подробный экран подключения.
- Настройки порта и имени хоста в UI.
- Реконнект после временного разрыва.
- Синхронизация погоды, если она появится.
- Серверная модель прав: host/admin/guest.
- Минимальные интеграционные тесты протокола без запуска OpenGL.
- Совместимость сохранений между версиями протокола.
- Документация формата пакетов для разработчиков.

## Не планируется для ближайших snapshot

- Публичные аккаунты и авторизация.
- Официальные публичные серверы.
- Relay/NAT traversal как встроенный сервис.
- Полноценный anti-cheat уровня публичных серверов.
- Полная совместимость с Minecraft blockstate/NBT.
- Redstone.
- Nether/End измерения.
- Импорт внешних schematic-файлов.
- Большой переписанный движок вместо текущей инкрементальной разработки.
